# Plan: Admin Dashboard OAuth Login Migration

> Status: Draft
> Created: 2026-03-29

## Goal

Convert admin dashboard from direct session auth (`POST /api/auth/login` + `admin_session` cookie) to OAuth 2.1 client that authenticates via the IdP's own login page (`apps/web`). This is the "eat your own dog food" pattern used by Auth0, Okta, Keycloak, and Google Workspace for their admin consoles.

## Current State

```
apps/admin                                  apps/server
────────────                                ────────────
/login page (own login form)
  │
  POST /api/auth/login ──────────────────►  validates credentials
  ◄────────────────────────────────────────  sets admin_session cookie
  │
  middleware checks admin_session cookie
  │
  layout.tsx fetches /api/account/profile
  checks role === 'admin' | 'super_admin'
```

Key files:
- `apps/admin/src/app/login/page.tsx` — login form (to be removed)
- `apps/admin/src/middleware.ts` — checks `admin_session` cookie, injects `x-session-cookie` header
- `apps/admin/src/app/(dashboard)/layout.tsx` — fetches profile, verifies admin role
- `apps/server/src/core/plugins/auth.ts` — `getSessionCookieName()` reads `x-session-cookie` header
- `packages/db/src/schema/oauth-client.ts` — no `isFirstParty` field yet
- `apps/server/src/modules/oauth/oauth.service.ts` — consent check, no auto-approve logic

## Target State

```
apps/admin                    apps/web (IdP UI)             apps/server
────────────                  ─────────────────             ────────────

User visits admin
  │
  middleware: no session?
  ──► redirect to /auth/login
  │
  /auth/login builds OAuth URL
  ──► redirect to idp/oauth/authorize ──►
                                          requireSession
                                          no session? ──► /login
                                          user logs in
                                          consent auto-approved
                                          (first-party client)
  ◄── redirect to /auth/callback?code=...
  │
  /auth/callback exchanges code
  ──► POST /oauth/token ─────────────────────────────────► validates
  ◄────────────────────────────────────────────────────── {access_token,
  │                                                        id_token,
  sets admin_session cookie                                refresh_token}
  (stores tokens server-side)
  │
  middleware: has session ──► allow
  │
  API calls use access_token
  via serverFetch Authorization header
```

### Step 2: Auto-approve consent for first-party clients

**Files to modify:**

- `apps/server/src/modules/client/client.schemas.ts` — add `isFirstParty` to response schema

**Changes in `authorize()`:**

Current logic (simplified):
```
1. Load client
2. Validate redirect_uri, scope, grant_type
3. Check consentGrants table — has user granted all requested scopes?
4. If not → return { type: 'consent_required' }
5. If yes → issue auth code, return { type: 'redirect' }
```

New logic — add check after step 2:
```
3. If client.isFirstParty → skip consent check entirely
   → issue auth code, return { type: 'redirect' }
4. (existing) Check consentGrants table...
```

This means first-party clients never show a consent screen. The user logs in and is immediately redirected back with an auth code.

### Step 4: Add OAuth callback route to admin app

**Files to create:**
- `apps/admin/src/app/auth/login/route.ts` — initiates OAuth flow
- `apps/admin/src/app/auth/callback/route.ts` — handles OAuth callback
- `apps/admin/src/lib/oauth.ts` — PKCE helpers and OAuth config

**`/auth/login` route handler (GET):**
```
1. Generate code_verifier (random 43-128 chars)
2. Compute code_challenge = base64url(sha256(code_verifier))
3. Generate random state
4. Store code_verifier + state in httpOnly cookie (short-lived, 10 min)
5. Redirect to:
   {IDP_URL}/oauth/authorize
     ?response_type=code
     &client_id=admin-dashboard
     &redirect_uri=http://localhost:3002/auth/callback
     &scope=openid profile email
     &code_challenge={challenge}
     &code_challenge_method=S256
     &state={state}
```

**`/auth/callback` route handler (GET):**
```
1. Read code and state from query params
2. Read code_verifier and expected state from cookie
3. Verify state matches (CSRF protection)
4. Exchange code for tokens:
   POST {API_URL}/oauth/token
     grant_type=authorization_code
     code={code}
     redirect_uri=http://localhost:3002/auth/callback
     code_verifier={code_verifier}
     client_id=admin-dashboard
     client_secret={ADMIN_CLIENT_SECRET}
5. Validate id_token (verify signature via JWKS, check iss, aud, exp, nonce)
6. Extract user info from id_token (sub, email, name)
7. Store tokens server-side:
   - Option A: encrypted cookie containing access_token + refresh_token
   - Option B: server-side session store (Map/Redis) keyed by session ID
8. Set admin_session cookie (httpOnly, secure, sameSite=lax)
9. Clear the PKCE cookie
10. Redirect to /users (admin home)
```

**`oauth.ts` helper:**
```
- generateCodeVerifier(): string
- generateCodeChallenge(verifier: string): string
- generateState(): string
- OAUTH_CONFIG: { clientId, clientSecret, issuer, redirectUri, scopes }
```

### Step 6: Update admin `serverFetch` to use access tokens

**Files to modify:**
- `apps/admin/src/lib/` — add token retrieval helper
- Or modify how `serverFetch` gets the Authorization header

**Current `serverFetch` pattern:**
```typescript
// Reads admin_session cookie value
// Sends as: Authorization: Bearer {cookie_value}
// Also sends: x-session-cookie: admin_session
```

**New pattern:**
```typescript
// Reads admin_session cookie → decrypts → gets access_token
// Sends as: Authorization: Bearer {access_token}
// No x-session-cookie header needed
```

Admin API calls now use OAuth access tokens instead of raw session tokens. The server validates these as JWT access tokens, not session lookups.

**Important:** Admin endpoints (`/api/admin/*`) currently use `requireAdmin` middleware which checks session → user → isAdmin role. This needs to also work with OAuth access tokens. See Step 7.

### Step 7: Admin API authorization via access token

**Files to modify:**
- `apps/server/src/core/plugins/auth.ts` — `requireSession` and `requireAdmin`

**Current flow:**
```
requireSession:
  1. Read admin_session cookie or Bearer token
  2. Look up session in DB → get userId
  3. Set request.userId

requireAdmin:
  1. (after requireSession)
  2. Look up user roles
  3. Check role includes 'admin' or has 'admin:access' permission
```

**Decision point — two approaches:**

**Approach A: Admin endpoints accept both session tokens AND OAuth access tokens**
- `requireSession` already reads Bearer tokens
- Add JWT verification fallback: if Bearer token is a JWT (has dots), verify as access token instead of session lookup
- Extract `sub` claim as userId
- This is the minimal change approach

**Approach B: Admin endpoints require OAuth access tokens with admin scope**
- Add admin-specific scopes: `admin:access`, `admin:users:read`, `admin:users:write`, etc.
- Token carries scopes → server checks scopes instead of looking up roles
- More OAuth-pure but bigger change

**Recommendation: Approach A for this migration.** Keep the existing RBAC system. The access token identifies the user (via `sub`), then existing role checks verify admin permissions. This minimizes server-side changes.

**Changes to `requireSession` in auth.ts:**
```
1. Read Bearer token from Authorization header
2. If token looks like a JWT (contains two dots):
   a. Verify JWT signature via JWKS
   b. Check iss, aud, exp claims
   c. Extract sub → set request.userId
   d. Skip session DB lookup
3. Else (opaque session token):
   a. Existing session lookup logic
```

### Step 8: Add token refresh logic

**Files to create/modify:**
- `apps/admin/src/app/auth/refresh/route.ts` — or handle inline in middleware

**When access token expires:**
```
1. Middleware or serverFetch detects 401 response
2. Read refresh_token from admin_session cookie
3. POST /oauth/token
     grant_type=refresh_token
     refresh_token={refresh_token}
     client_id=admin-dashboard
     client_secret={ADMIN_CLIENT_SECRET}
4. Receive new access_token + refresh_token
5. Update admin_session cookie with new tokens
6. Retry original request
```

**Edge case:** If refresh also fails (token revoked, expired), redirect to `/auth/login` to re-authenticate.

### Step 9: Update admin logout

**Files to modify:**
- `apps/admin/src/app/(dashboard)/layout.tsx` or wherever logout is triggered
- Create `apps/admin/src/app/auth/logout/route.ts`

**Current logout:**
```
POST /api/auth/logout → clears admin_session cookie
```

**New logout:**
```
1. Read refresh_token from admin_session cookie
2. POST /oauth/revoke { token: refresh_token }  ← revoke refresh token
3. Clear admin_session cookie
4. Redirect to IdP end-session:
   GET /oauth/end-session
     ?id_token_hint={id_token}
     &post_logout_redirect_uri=http://localhost:3002/auth/login
5. IdP revokes session, redirects back to admin login
```

## Architecture Decision Records

### Why not admin-specific scopes?

We could encode admin permissions as OAuth scopes (`admin:users:read`, `admin:sessions:write`, etc.) and check them in the token. We chose NOT to because:
- The RBAC system already exists and works
- Scopes would duplicate the role-permission model
- Changing permissions would require re-issuing tokens
- The access token identifies the user; the role system authorizes actions

### Why encrypted cookie, not server-side session store?

Storing tokens in an encrypted httpOnly cookie avoids adding another stateful store (Redis/Map). The cookie contains `{ access_token, refresh_token, id_token }` encrypted with `ADMIN_SESSION_SECRET`. Trade-off: cookie size is larger (~1-2KB) but well within the 4KB browser limit.

### Why PKCE even though admin is confidential?

OAuth 2.1 mandates PKCE for ALL clients, including confidential ones. Even though admin has a client_secret, PKCE prevents authorization code injection attacks in the redirect.
